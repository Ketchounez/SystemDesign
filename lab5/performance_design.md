# Лабораторная работа №5

## 1. Анализ производительности

Вариант 13 — система бронирования отелей с сущностями `User`, `Hotel`, `Booking`.

Hot paths:
- `GET /api/v1/hotels` и `GET /api/v1/hotels?city=...`, так как список отелей часто просматривается.
- `GET /api/v1/users/{user_id}/bookings`, так как пользователь регулярно открывает активные и прошлые бронирования.
- `POST /api/v1/bookings` при всплесках нагрузки (массовые попытки бронирования).

Потенциально медленные операции:
- чтение списка отелей из PostgreSQL;
- чтение бронирований пользователя из MongoDB;
- проверка конфликтов дат при создании бронирования (поиск пересечений среди активных броней отеля).

Целевые требования для данной лабораторной:
- P95 задержки для кешируемых read-endpoint ниже 50 мс в локальной среде;
- стабильная работа при всплесках запросов на создание бронирования;
- деградация без отказа сервиса под высокой нагрузкой (возврат `429`, а не перегрузка БД).

## 2. Проектирование стратегии кеширования

Выбранная стратегия: **Cache-Aside (Lazy Loading)** с локальным кешем в сервисе; rate limiting реализован через Redis.

Кешируемые endpoint:
1. `GET /api/v1/hotels` и `GET /api/v1/hotels?city=...`
   - ключ: `all` или `city:{normalized_city}`;
   - TTL: 60 секунд;
   - инвалидация: полный сброс кеша отелей при `POST /api/v1/hotels`.
2. `GET /api/v1/users/{user_id}/bookings`
   - ключ: `user_id`;
   - TTL: 30 секунд;
   - инвалидация: сброс кеша конкретного пользователя при `POST /api/v1/bookings` и `DELETE /api/v1/bookings/{booking_id}`.

Обоснование:
- данные читаются значительно чаще, чем изменяются;
- короткий TTL сохраняет баланс между актуальностью и разгрузкой БД;
- явная инвалидация после записи предотвращает устаревшие ответы.

## 3. Проектирование rate limiting

Защищаемый endpoint: `POST /api/v1/bookings` (наиболее критичен при всплесках нагрузки).

Алгоритм: **Fixed Window Counter** для каждого аутентифицированного пользователя.

Параметры:
- лимит: 20 запросов за 1 минуту на пользователя;
- сброс окна: один раз в минуту от первого запроса в текущем окне.

Поведение API:
- при допустимом запросе: стандартная обработка + заголовки лимитов;
- при превышении: HTTP `429 Too Many Requests` + JSON-ошибка + заголовки лимитов;
- заголовки:
  - `X-RateLimit-Limit`
  - `X-RateLimit-Remaining`
  - `X-RateLimit-Reset` (unix timestamp).

Примечание по масштабированию:
- в текущей реализации счетчик лимитов хранится в Redis, что позволяет разделять лимиты между инстансами;
- для production-сценариев это снижает риск расхождения лимитов при горизонтальном масштабировании.

## 4. Влияние на производительность и мониторинг

Ожидаемый эффект:
- снижение нагрузки на БД за счет кеш-хитов на частых чтениях;
- снижение пикового давления на запись за счет ограничения частоты `POST /api/v1/bookings`;
- более предсказуемое поведение сервиса при нагрузочных всплесках.

Метрики для мониторинга:
- перцентили HTTP-задержек (P50/P95/P99) по endpoint;
- пропускная способность (RPS);
- количество и задержка запросов к PostgreSQL/MongoDB;
- метрики кеша:
  - hit rate кеша отелей;
  - hit rate кеша бронирований пользователя;
- метрики rate limiter:
  - количество разрешенных запросов;
  - количество отклоненных запросов (`429`).

Как измерять эффективность кеширования:
- формула hit rate: `hits / (hits + misses)`;
- сравнение количества обращений к БД до и после включения кеша при одинаковом синтетическом профиле нагрузки;
- проверка корреляции: рост hit rate должен сопровождаться снижением средней задержки и P95.
